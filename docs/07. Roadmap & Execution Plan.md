### Langkah 1.2 — Metric Conflation Fix (1 hari) ✅

**Error rate:** 4-5% dari total claims | **Status: ✅ Selesai (commit `2993cf0`)**

**Akar masalah:** Model nulis metric name yang salah. Angka bener, label metriknya yang nyasar.
```
Komparasi: "comparable store sales" ≠ "total revenue"
Di source: "total revenue increased 5.2%"
Di jawaban: "comparable store sales increased 5.2%"   ❌
```

**Role Discussion:**

**CEO:** Ini lebih subtle dari number transposition — user yang gak familiar sama istilah finansial mungkin gak notice. Tapi hiring manager yang dari finance bakal langsung lihat "ini orang gak ngerti bedanya comparable store sales sama total revenue". Yang bikin ini penting: error ini nunjukin kalo model gak bener-benar "mengerti" konteks finansial.

**Engineer:** Tantangannya di synonym resolution. Model mungkin jawab "same-store sales" tapi source bilang "comparable store sales" — itu sinonim, bukan error. Kita udah punya 20 synonym group di `query_expansion.py`, tinggal复用. Perlu bikin `MetricVerifier` class baru.

**Designer:** UX: kalo metric mismatch, tampilin tooltip "The metric name in the answer differs from the source document. Source says: [exact metric]."

**Programmer:**

```python
# Step 1 — Build MetricVerifier class di src/post_process.py

class MetricVerifier:
    """
    Verifies that metric names in LLM answer match metric tags in source chunks.
    Synonym-aware — uses 20 groups from query_expansion.py.
    """
    
    def __init__(self, synonym_groups: dict[str, list[str]] = None):
        # Default: 20 groups dari GLOSSARY_GROUPS
        from query_expansion import GLOSSARY_GROUPS
        self.synonym_groups = synonym_groups or GLOSSARY_GROUPS
        # Build reverse mapping: "comparable sales" → "comparable_store_sales"
        self._term_to_group = {}
        for group_name, terms in self.synonym_groups.items():
            for term in terms:
                self._term_to_group[term.lower()] = group_name
    
    def extract_metrics(self, text: str) -> list[str]:
        """Extract metric names from text using _METRIC_TRIGGERS + synonym lookup."""
        from post_process import _METRIC_TRIGGERS
        found = []
        for trigger in _METRIC_TRIGGERS:
            if trigger.lower() in text.lower():
                found.append(trigger)
        return found
    
    def resolve_group(self, metric_name: str) -> str | None:
        """Resolve a metric name to its synonym group."""
        return self._term_to_group.get(metric_name.lower())
    
    def verify(self, answer: str, source_chunks: list[dict]) -> list[dict]:
        """
        Cross-check: metric di jawaban harus match metric di minimal 1 source chunk.
        Returns list of mismatches.
        """
        answer_metrics = self.extract_metrics(answer)
        source_metrics = set()
        for chunk in source_chunks:
            source_metrics.update(self.extract_metrics(chunk.get("text", "")))
        
        # Resolve both to synonym groups
        answer_groups = set()
        for m in answer_metrics:
            g = self.resolve_group(m)
            if g:
                answer_groups.add(g)
        
        source_groups = set()
        for m in source_metrics:
            g = self.resolve_group(m)
            if g:
                source_groups.add(g)
        
        mismatches = []
        for m in answer_metrics:
            g = self.resolve_group(m)
            if g and g not in source_groups:
                mismatches.append({
                    "metric": m,
                    "group": g,
                    "source_metrics": list(source_metrics)
                })
        
        return mismatches

# Step 2 — Integrasi ke src/rag.py (bareng sama number verifier)
verifier = MetricVerifier()
metric_issues = verifier.verify(answer_text, context_chunks)
if metric_issues:
    logger.warning(f"[MetricVerifier] {len(metric_issues)} metric mismatch(es)")

# Step 3 — Test false positive rate
# Jalankan di 20 eval questions
# Target: false positive < 10%
# Kalo >10%: tambah synonym group

# Files:
#   src/post_process.py — +~80 lines (MetricVerifier)
#   src/rag.py — +5 lines (integrasi)

# Edge cases:
# - "revenue" vs "total revenue": harusnya match (sinonim)
# - "same-store sales" vs "comparable store sales": match
# - "comparable store sales" vs "total revenue": mismatch
```

**Business:** Risiko false positive. Kalo terlalu sering flag "revenue" vs "total revenue" sebagai error, user jadi desensitized. Target: flag cuma ketika metric bener-bener beda group (comparable sales ≠ revenue).

### Langkah 1.3 — Causal Proximity Fix (2-3 hari) ✅

**Error rate:** ~7% dari total claims | **Status: ✅ Selesai (commits `51e0093`, `23b5704`)**

**Akar masalah:** Bukan generation error, tapi **retrieval error**. Model "nyasar" ke chunk tetangga yang beda metrik.
```
Kasus: Tanya labor costs → retrieve chunk yang bahas labor costs
       TAPI di chunk itu juga ada paragraph soal food costs (berdekatan di filing)
       → LLM baca dua-duanya → jawaban nyebut food cost driver    ❌
```

**Role Discussion:**

**CEO:** Ini error paling subtle — jawabannya keliatan bener, angka cocok, tapi konteksnya salah. Bahaya karena user yang gak domain expert gak bakal notice. Fix ini yang paling nambah credibility teknis.

**Engineer:** Ini error di layer retrieval, bukan generation. Fix-nya di hulu — sebelum cross-encoder. Pendekatan: tag tiap chunk dengan metric label pas ingestion. Pas retrieval, filter chunks yang metric_label-nya match sama query. Dua pendekatan: keyword-based (~80% akurasi, 50 lines) atau LLM-based (~95%, 432 LLM calls). Rekomendasi: keyword-based dulu.

**Programmer:**

```python
# Step 1 — Bangun metric tagging function

from query_expansion import GLOSSARY_GROUPS

def tag_chunk(chunk_text: str, chunk_metadata: dict) -> dict:
    """
    Tag a chunk with financial metric labels.
    Returns enriched metadata with metric + metric_keywords.
    """
    text_lower = chunk_text.lower()
    
    metric_scores = {}
    for group_name, keywords in GLOSSARY_GROUPS.items():
        score = sum(1 for kw in keywords if kw.lower() in text_lower)
        if score >= 2:  # threshold: min 2 keyword match
            metric_scores[group_name] = score
    
    # Sort by score descending, pick top
    sorted_metrics = sorted(metric_scores.items(), key=lambda x: -x[1])
    
    enriched = dict(chunk_metadata)
    if sorted_metrics:
        enriched["metric"] = sorted_metrics[0][0]
        enriched["metric_keywords"] = GLOSSARY_GROUPS[sorted_metrics[0][0]]
        enriched["metric_confidence"] = sorted_metrics[0][1]
    else:
        enriched["metric"] = "general"
        enriched["metric_keywords"] = []
        enriched["metric_confidence"] = 0
    
    return enriched

# Di retrieval.py, di dalam query_multi():
# SEBELUM cross-encoder, filter chunks by metric

def _filter_by_metric(chunks: list[dict], query: str) -> list[dict]:
    """Filter chunks that match query metric."""
    # Extract metric dari query
    query_metric_hints = []
    from post_process import _METRIC_TRIGGERS
    for trigger in _METRIC_TRIGGERS:
        if trigger.lower() in query.lower():
            query_metric_hints.append(trigger)
    
    if not query_metric_hints:
        return chunks  # No metric detected in query — return all
    
    # Filter chunks: retain yang metric_label match minimal 1 query hint
    # Gak strict — kalo hasil kosong, return original chunks
    filtered = []
    for chunk in chunks:
        chunk_metric = chunk.get("metadata", {}).get("metric", "")
        for hint in query_metric_hints:
            if hint.lower() in chunk_metric.lower():
                filtered.append(chunk)
                break
    
    # Graceful degradation: kalo filtered kosong, return original
    return filtered if filtered else chunks
```

## Phase B — Platform Evolution

> **Research basis:** Analysis of 7 trending repos + financial RAG papers (FinAgent-RAG, HierFinRAG, LedgerRAG, BAM Embeddings). Documented in `_analysis/09. Improvement Blueprint from Trending Repos`.
>
> **6-role review:** Engineer (feasibility), Programmer (complexity), Designer (UX gap), Business (portfolio value), Security (data isolation), CEO (effort vs impact).

### P0 — Multi-turn Conversation + Memory

**The gap:** Single-query only. Analysts need follow-ups and session continuity.

**Approach:**
- Session management API: `POST /sessions`, `GET /sessions/{id}/messages`
- Summary-based memory + sliding window (5-10 recent turns, periodic LLM summarization)
- Query rewriting for co-reference resolution ("it", "that" → financial entities)
- Token budget management (trigger compression at 80% context window)
- Immutable append-only history with fork/rollback semantics

**References:** LangChain SummaryBufferMemory, LlamaIndex ChatSummaryMemoryBuffer, tianpan.co (2026) session-as-resource design, LangGraph checkpointer.

### P0 — Better Number & Table Understanding

**The gap:** 38.8% of financial QA errors are arithmetic (FinAgent-RAG). Tables get split across chunks.

**Approach:**
- **Program-of-Thought (PoT)** — generate Python code for arithmetic, sandbox-execute, paste result. Reduces arithmetic errors 88%.
- **Table-aware chunking** — merge consecutive table-row chunks (max 2× size), linearize with header-prepended rows
- **Numeric verifier** — post-generation number extraction + recalculation from sources
- **Metadata-driven retrieval** — tag chunks by metric type, filter during hybrid search

**References:** FinAgent-RAG (arXiv 2605.05409), HierFinRAG (MDPI 2026, 82.5% FinQA), MimirRAG (arXiv 2605.25030), LedgerRAG (Zenodo 2026).

### P1 — Fine-tuned Embedding Model

**The gap:** Generic nomic-embed confuses financial terms ("par value", "CAGR").

**Approach:**
- **Matryoshka Representation Learning** — fine-tune on philschmid/financial-rag-embedding-dataset (~7k Q&A pairs). One model: 768/512/256/128/64 dims.
- **Multi-dim cascade** — 128d fast path → cross-encoder → 768d escalation if low confidence
- **Training:** Sentence Transformers 3, MNRL + MatryoshkaLoss, ~3 min on consumer GPU

**References:** philschmid.de fine-tune guide, rya23/modernbert-embed-finance-matryoshka (+10% NDCG@10), BAM Embeddings (62.8% R@1 vs 39.2% OpenAI), shail-2512/nomic-embed-financial-matryoshka.

### P1 — Docker + Production Readiness

**The gap:** Manual venv setup. No one-click deploy. Streamlit only local.

**Approach:**
- Docker Compose: Ollama + ChromaDB + FastAPI + Streamlit + nginx
- Hosted demo with rate limiting + session data isolation
- PDF/Word report export with inline citations

**References:** NithiN-1808/agentic-financial-rag (Docker Compose stack), Streamlit Cloud / Railway.

### P2 — Evaluation & Experimentation

**The gap:** Custom faithfulness eval. No RAGAS. LLM-as-Judge unvalidated.

**Approach:**
